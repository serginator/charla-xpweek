Clean Code

Principios básicos


  • Código fácil de leer, expresivo, sencillo
  • Hace una única cosa
  • Sin duplicidad
  • Probado mediante tests automáticos
  • Abstrae elementos similares
  • Usa el menor número de elementos posibles
  • Código mantenible

Nombres

Buenos nombres, descriptivos y claros

                        var a, //más info...
colorcito, //poco descrpitivo
bsa; //no me dice nada
CACA 
                    var x, // matemáticas, ok
azul, // sin dudas
borrarSlideAnterior; //bien
BIEN

Funciones

Se puede inferir su comportamiento de un vistazo.
  • Cortas
  • Hacen una única cosa
  • Usar buenos nombres
  • No pasar booleanos como parámetros (hacen que la función haga como mínimo dos cosas) 
                    var x = function(a, b) {
          return a + b;
}
CACA 
                var suma = function(a, b) {
          return a + b;
}
BIEN

Comentarios

  • Si el código está bien escrito debería ser autoexplicativo
  • Para explicar el algoritmo utilizado
  • TODOs
  • Avisar de algo, Warnings.
  • JavaDoc para generar documentación

Formateo

  • Legibilidad del código
  • Seguir unas guías de estilo (más hincapié sobre todo en trabajo en equipo)
  • Da igual cuáles sean, con tal de que todos las sigan
  • Da uniformidad al código
  • Dependencia Top-Down (cuidado con el hoisting)

Objetos/Estructuras

Hay que pensar en qué es necesario utilizar, no tienen que ser siempre objetos o siempre estructuras de datos.


Demeter's Law: Un método f de una clase C debería llamar sólo a:
  • Métodos de C
  • Métodos de objetos creados con f
  • Métodos de objetos pasados como argumentos a f
  • Campos y propiedades del propio objeto

Manejo de Errores

  • Usar Excepciones que definan bien el error en lugar de códigos de retorno
  • Evitar pasar/devolver null, así prevenimos comprobaciones con null.

Fronteras

Definir de forma clara y concisa la separación entre nuestro código y el código de terceros (librerías, componentes...)

TDD


  1. No se escribe código de producción hasta que no haya un test fallido que arreglar
  2. No se debe escribir más de un test para que éste falle
  3. No se escribe más código de producción del necesario para pasar el test.

TDD (2)

  • Mantener los tests limpios
  • Permiten lanzarse a refactorizar sin miedo
  • Un assert (o los mínimos necesarios) por test, que va directamente relacionado con...
  • Un único concepto por test

TDD (3)

Las reglas FIRST
  • Fast: ejecución rápida y a menudo
  • Independent: Un test no debe depender de uno anterior
  • Repeteable: Se deben poder realizar en cualquier entorno
  • Self-Validating: El test debe decir si se cumple o no
  • Timely: Escribirlos en el momento adecuado, justo antes del de producción

Clases

  • Deben ser pequeñas
  • Principio de responsabilidad única: "Una clase debe tener una y solo una única causa por la cual puede ser modificada"
  • Fácil de cambiar

Sistemas

  • Reconocer las distintas responsabilidades del sistema.
  • Empezar por lo pequeño
  • Añadir features después

Emergencia

4 reglas de Kent Beck
  1. Ejecutar TODOS los tests
  2. Eliminar la duplicidad
  3. Expresar la intención del programador
  4. Minimizar número de clases y métodos

Concurrencia

  • Mejora el rendimiento
  • Es compleja de manejar
  • Mantener los bloques pequeños
  • No compartir objetos (podemos liarla fina)
  • Errores dificilmente reproducibles...

En resumen

  • Empezar con código sucio e ir limpiándolo
  • Mantenerlo todo pequeño
  • Mantener código legible
  • Autoexplicativo
  • Tests
  • Interfaces, clases abstractas, OO

NPM

  • npm init -> genera un package.json inicial

Grunt Wildcards

  • ? -> una letra
  • * -> 0 o más letras
  • ** -> cualquier cosa (ej: src/**/*.js)
  • ! -> menos (ej: src/**/!(experimento_*).js)
  • {} -> uno u otro (ej: src/**/*.{js,html})
